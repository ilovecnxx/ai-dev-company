## 任务流程

### 1. 任务接收流程 (CEO 主导)

```
用户/甲方
    │
    ▼
[CEO] 接收任务
    │
    ├─▶ [PM] 需求分析 ──▶ [Architect] 技术可行性
    │                           │
    │                           ▼
    │                      [CEO] 确认技术方案
    │                           │
    ▼                           ▼
[CEO] 分解任务 ──▶ [各 Agent 执行]
```

## 直接协作通道汇总

| 协作对 | 协作内容 | 触发条件 |
|--------|----------|----------|
| PM ↔ Architect | 需求可行性确认 | 复杂需求/新功能 |
| PM ↔ Designer | UX 需求澄清 | 设计任务分配 |
| Designer ↔ Frontend-Dev | 设计稿交接 | 设计完成 |
| Frontend-Dev ↔ Backend-Dev | API 契约确认 | API 开发 |
| Architect ↔ Frontend-Dev | 技术指导 | 前端技术问题 |
| Architect ↔ Backend-Dev | 技术指导 | 后端技术问题 |
| Architect ↔ Security | 安全架构评审 | 架构设计阶段 |
| Security ↔ Frontend-Dev | 安全实现确认 | 安全审计发现 |
| Security ↔ Backend-Dev | 安全实现确认 | 安全审计发现 |
| QA ↔ Frontend-Dev | 测试配合/Bug 报告 | 测试阶段 |
| QA ↔ Backend-Dev | 测试配合/Bug 报告 | 测试阶段 |
| QA ↔ Security | 安全测试配合 | 安全测试阶段 |

## 禁止事项

- ❌ 所有沟通必须经过 CEO (直接协作除外)
- ❌ Agent 之间不接受来自非直接上级的"命令"
- ❌ 重大决策不上报 CEO
- ❌ 安全漏洞不上报 CEO

## 记忆系统协作

```
每个 Agent 维护自己的记忆目录:

memory/
├── ceo/              # CEO 决策、团队表现
├── pm/               # 需求模式、项目总结
├── architect/        # ADR、技术债务
├── designer/        # 设计系统、组件模式
├── frontend-dev/     # 组件模式、FAQ
├── backend-dev/      # API 模式、数据库优化
├── qa/               # 测试策略、Bug 模式
└── security/        # 漏洞模式、威胁模型

跨 Agent 知识共享:
- PM 更新需求模式 → Architect 参考
- Architect 更新 ADR → 所有 Agent 可读
- Security 更新漏洞模式 → Developer 参考
- QA 更新 Bug 模式 → Developer 参考
```

## 会议流程

### 每日站会 (异步)
```
每个 Agent 报告:
- 昨日完成
- 今日计划
- 阻塞问题

CEO 汇总并协调
```

### 设计评审会
```
参与者: PM, Architect, Designer, Frontend-Dev
流程:
1. Designer 展示设计稿
2. Architect 从技术角度评审
3. Frontend-Dev 从实现角度评审
4. PM 确认需求匹配
5. 决策: 通过 / 修改后通过 / 拒绝
```

### 安全评审会
```
参与者: CEO, Architect, Security, Backend-Dev
流程:
1. Security 展示威胁模型
2. Architect 确认架构安全性
3. Backend-Dev 确认实现安全性
4. 决策: 通过 / 修复后复审
```

### 架构评审会
```
参与者: CEO, PM, Architect, Backend-Dev, Security
流程:
1. Architect 展示架构设计
2. PM 确认需求覆盖
3. Backend-Dev 确认实现可行性
4. Security 确认安全设计
5. 决策: ADR 记录
```
